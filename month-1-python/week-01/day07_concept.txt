Concept 1 -  A string is an immutable sequence of characters

name = "Karthik"
#       0123456
print(name[0])    # 'K'  — first character
print(name[6])    # 'k'  — last character (length is 7, so last index is 6)
print(name[-1])   # 'k'  — negative counts from the right
print(name[-2])   # 'h'

So a string behaves like the lists you learned on Day 5: you index with [], positions start at 0, and negative indices walk backward from the end. Same indexing model, different type.
Slicing works the same way too — s[start:stop] where stop is exclusive:

word = "engineer"
#       01234567
print(word[0:4])   # 'engi'  — indices 0,1,2,3 (NOT 4)
print(word[:4])    # 'engi'  — omitting start means "from the beginning"
print(word[4:])    # 'neer'  — omitting stop means "through the end"
print(word[::-1])  # 'reenigne'  — step of -1 reverses it

A slice has three parts, not two: s[start:stop:step]. So far you've only used two (start:stop) and the step silently defaulted to 1 — meaning "walk forward one character at a time." The third number changes that stride.

lang = "Python"
#       012345

Start omitted → begin at the front. Stop omitted → go to the end. Step 2 → take every second character. So it visits index 0, then 2, then 4:

index:  0    2    4
char:   P    t    o

Result: "Pto". (It skips indices 1, 3, 5 → y, h, n.)

Negative step — lang[::-1]:

A negative step means walk backward. Here's the subtle part: when the step is negative, the defaults flip. An omitted start now means "begin at the end," and an omitted stop means "go to the beginning." So [::-1] says: start at the last character, walk backward one at a time, until you fall off the front:

index:  5    4    3    2    1    0
char:   n    o    h    t    y    P

Result: "nohtyP". That's why it reverses — it's not a special "reverse" operation, it's just a normal slice reading right-to-left.

Here's the one thing that separates strings from lists — strings are immutable. You cannot change a character in place:

name = "Karthik"
name[0] = "P"   # TypeError: 'str' object does not support item assignment

This connects directly to something you flagged on Day 5. Remember how .sort() mutates a list and returns None? String methods are the opposite: because a string can't be mutated, every method that "changes" a string actually returns a brand-new string and leaves the original untouched:

greeting = "hello"
loud = greeting.upper()
print(loud)       # 'HELLO'  — the new string
print(greeting)   # 'hello'  — original is unchanged

if you ever write greeting.upper() and expect greeting itself to change, nothing happens — you have to capture the return value. That's the mirror image of the list-mutation trap.


Concept 2 — Case and whitespace methods

The most common cleanup you'll do on messy text — user input, scraped data, file contents — is fixing case and stripping stray whitespace.

raw = "  Hello World  "

print(raw.upper())      # '  HELLO WORLD  '  — every letter uppercased
print(raw.lower())      # '  hello world  '  — every letter lowercased
print(raw.strip())      # 'Hello World'      — whitespace removed from BOTH ends
print(raw)              # '  Hello World  '  — original untouched, as always

Three things worth pinning down about .strip():
It only removes from the ends, never the middle — "a b c".strip() is still "a b c" because the spaces are interior. It removes all whitespace by default: spaces, tabs (\t), newlines (\n). And it has one-sided cousins, .lstrip() (left only) and .rstrip() (right only).

A close relative of case methods is .title(), which capitalizes the first letter of each word:
print("ai engineer journey".title())   # 'Ai Engineer Journey'

Notice it capitalized "Ai" — .title() is naive about acronyms. Good to know before you trust it on real data.

Concept 3 — Searching and testing

To ask "does this string contain X?" or "does it start/end with X?":

email = "karthik@example.com"

print("@" in email)              # True   — the `in` operator, same as with lists
print(email.startswith("kar"))   # True
print(email.endswith(".com"))    # True
print(email.endswith(".org"))    # False

print(email.find("@"))           # 7      — index where "@" first appears
print(email.find("z"))           # -1     — NOT found returns -1 (does not raise)

That .find() returning -1 instead of raising is a deliberate design choice — it's the string equivalent of dict .get() returning None instead of a KeyError. Same philosophy: "not found" is a value, not an error. (There's a stricter sibling .index() that does raise, mirroring the .remove() vs .discard() split you saw with sets.)

Concept 4 — .split() and .join(), the two you'll live in

These are the workhorses. In ML and GenAI work you're constantly turning a string into a list of pieces, processing the pieces, then stitching them back into a string. .split() does the first, .join() does the second — they're inverses.
.split() — string → list:

sentence = "the quick brown fox"
words = sentence.split()          # split on whitespace by default
print(words)                      # ['the', 'quick', 'brown', 'fox']
print(len(words))                 # 4

Called with no argument, it splits on any run of whitespace. But you can pass a separator to split on something specific:

csv_row = "karthik,berlin,engineer"
fields = csv_row.split(",")
print(fields)                     # ['karthik', 'berlin', 'engineer']

date = "2026-07-04"
print(date.split("-"))            # ['2026', '07', '04']

The separator you pass is consumed — it doesn't appear in the output pieces.

.join() — list → string:

This one has surprising syntax that trips almost everyone, so look closely. The separator is the string you call the method on, and the list goes inside the parentheses:

words = ['the', 'quick', 'brown', 'fox']
print(" ".join(words))            # 'the quick brown fox'   — joined with spaces
print("-".join(words))            # 'the-quick-brown-fox'   — joined with hyphens
print("".join(words))             # 'thequickbrownfox'      — joined with nothing


Read " ".join(words) as: "take this space, and use it as glue between every element of words." It feels backwards — you might expect words.join(" ") — but it's separator.join(list), not list.join(separator). This is a fresh version of the cross-language habit you flagged earlier (word.len() vs len(word)): your instinct will say words.join(...), and Python will reject it with an AttributeError because lists have no .join. The glue goes first.
One catch: .join() only works on a list of strings. If any element is a number, you get a TypeError — you'd have to convert them first. We'll leave that for later; just know the elements must already be strings.
